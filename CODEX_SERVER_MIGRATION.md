# Codex App-Server Migration Notes

## Purpose

This document onboards the next developer to the Codex app-server migration work completed in this repository on **March 6, 2026**.

The goal of the migration was to move the Codex provider path from a partially aligned / optimistic implementation to a **schema-verified `codex app-server` integration** that behaves correctly against the real JSON-RPC protocol exposed by the installed Codex CLI.

This is specifically about the Codex provider used by the MCP server in this repo, not about the entire provider stack.

---

## Scope of the Migration

### Why this work was needed

The repository already had a Codex provider path, but the app-server integration had drifted from the real protocol in several important places. The drift was not cosmetic; it affected correctness in areas such as:

- graceful turn interruption
- streamed output handling
- token accounting
- request-user-input responses
- MCP elicitation responses
- process cleanup
- documentation accuracy

The migration focused on bringing the implementation back into line with the actual `codex app-server` schemas generated by the installed CLI.

### Main outcomes

After this work:

- the Codex app-server client tracks and uses `turnId`
- interrupt requests are sent with both `threadId` and `turnId`
- token usage is read from the correct notification surface
- streamed text/output deltas are read from the correct fields
- camelCase thread item types are handled correctly
- `requestUserInput` responses are schema-valid
- MCP elicitation responses are schema-valid
- completed app-server processes are destroyed instead of being retained
- the Codex adapter reference doc now matches the implementation much more closely

## What Changed in the Implementation

## 1) `src/providers/codex-app-server.ts`

This file now behaves as the low-level app-server client should.

### Key changes

- Added active `turnId` tracking
  - captures turn ID from `turn/start` response and `turn/started`
- Updated graceful interrupt logic
  - `turn/interrupt` now includes both:
    - `threadId`
    - `turnId`
- Moved reasoning effort to the correct layer
  - reasoning effort is now sent per turn via `turn/start.effort`
  - it is no longer incorrectly attached to thread start/resume params
- Improved Codex error parsing
  - now tolerates the actual `codexErrorInfo` shapes better
  - supports both string and object forms more gracefully
- Kept backpressure retry handling for `-32001`
- Preserved process registration and teardown behavior

## 2) `src/providers/codex-adapter.ts`

This file now maps the real protocol much more accurately into the repo's provider/task abstractions.

### Key changes

- Updated event handling for real app-server item types
  - camelCase item types are now recognized correctly, including:
    - `agentMessage`
    - `commandExecution`
    - `fileChange`
    - `mcpToolCall`
    - `webSearch`
    - review items
- Updated delta handling
  - `item/agentMessage/delta` now reads `params.delta`
  - `item/commandExecution/outputDelta` now reads `params.delta`
- Updated token usage handling
  - now reads from `thread/tokenUsage/updated`
  - uses `tokenUsage.last.inputTokens` / `outputTokens`
- Updated turn-completion handling
  - now interprets `params.turn.status` and `params.turn.error`
  - no longer assumes the old top-level usage/error shape
- Fixed `requestUserInput` response shape
  - now returns:
    - `{ answers: { [questionId]: { answers: string[] } } }`
- Fixed MCP elicitation response
  - now returns a valid cancellation payload when unsupported
- Destroyed completed app-server clients on teardown
  - avoids retaining dead/finished app-server processes in `activeClients`
- Added small polish after the main migration
  - more readable `thread/status/changed` logging
  - less optimistic PATH-based app-server detection via `which` / `where`

## 3) Documentation

`docs/provider-abstraction/06-codex-adapter-reference.md` was updated to reflect the actual behavior, especially around:

- interrupt semantics
- token usage source
- request-user-input answer shape
- completion cleanup
- PATH detection behavior

## Primary Protocol Mismatches That Were Fixed

These were the main correctness issues found during the migration.

| Area | Previous problem | Current behavior |
|---|---|---|
| Turn interrupt | `turn/interrupt` was sent without `turnId` | Sends both `threadId` and `turnId` |
| Reasoning effort | Sent at thread start/resume in the wrong field | Sent per turn as `turn/start.effort` |
| Agent message delta | Looked for `params.text` | Reads `params.delta` |
| Command output delta | Looked for `params.output` | Reads `params.delta` |
| Item types | Expected underscore-style names | Handles real camelCase item types |
| Token usage | Assumed it came from `turn/completed` | Reads `thread/tokenUsage/updated` |
| Turn completion parsing | Assumed old top-level payload shape | Reads `params.turn.status` and `params.turn.error` |
| `requestUserInput` response | Returned invalid per-question shape | Returns `answers: string[]` per question |
| MCP elicitation response | Returned `{}` | Returns valid `{ action, content }` |
| Client lifecycle | Completed app-server clients were retained | Completed clients are destroyed and removed |

## Direct interrupt verification

A direct Node script was run against the built `CodexAppServerClient` to confirm that turn ID capture and interrupt handling actually work.

### Interrupt verification command

```bash
node --input-type=module - <<'NODE'
import { CodexAppServerClient } from './build/providers/codex-app-server.js';
const client = new CodexAppServerClient({});
const seen = [];
let completed = false;
try {
  await client.start();
  await client.startThread({ model: 'gpt-5.4', cwd: process.cwd(), sandboxMode: 'workspace-write', approvalPolicy: 'never' });
  const consume = (async () => {
    for await (const msg of client.runTurn('Write an extremely long response with at least 200 numbered bullet points, each with 2-3 sentences. Keep going until finished.', { reasoningEffort: 'medium' })) {
      seen.push(msg.method);
      if (msg.kind === 'notification' && msg.method === 'turn/completed') completed = true;
      if (seen.length >= 40) break;
    }
  })();
  const startedAt = Date.now();
  while (!client.currentTurnId && Date.now() - startedAt < 15000) {
    await new Promise(r => setTimeout(r, 100));
  }
  console.error('TURN_ID', client.currentTurnId ?? 'none');
  if (!client.currentTurnId) throw new Error('turnId was not captured before timeout');
  await new Promise(r => setTimeout(r, 1500));
  const interruptResult = await client.interruptTurn();
  console.error('INTERRUPT_ACK', interruptResult);
  await Promise.race([
    consume,
    new Promise((_, reject) => setTimeout(() => reject(new Error('consume timeout')), 30000)),
  ]);
  console.error('COMPLETED_SEEN', completed);
  console.error('METHODS', JSON.stringify(seen.slice(0, 20)));
} finally {
  client.destroy();
}
NODE
```

Observed outcome:

- `TURN_ID ...` was printed
- `INTERRUPT_ACK true`
- `COMPLETED_SEEN true`

This is the strongest direct runtime proof that the `turnId` fix works.

## Known Remaining Risks / Limitations

None of the below are current stop-ship blockers, but they are real follow-up items.

## 1) Multi-question `requestUserInput` is still best-effort

This is the most important remaining limitation.

### Current behavior

If Codex asks multiple questions in one `item/tool/requestUserInput` call, the adapter currently:

1. collapses them into one combined displayed prompt
2. collects one response through the existing `questionRegistry`
3. tries to split the response back into one answer per question

### Why this is acceptable today

- the response is schema-valid
- single-question flows are fine
- the current behavior is much better than returning an invalid payload

### Why this should still be improved next

This is not truly structured handling. It can mis-map answers when:

- multiple answers are ambiguous
- choices need to be preserved per-question
- order and formatting are unclear
- a future Codex flow leans harder on multi-question interactions

**This is the top next-step item.**

## 2) Unknown future server requests still use a generic fallback

In `handleAppServerRequest`, unknown methods still fall back generically. That is acceptable now, but a future protocol addition with a strict required response schema could need explicit support.

## 3) Observability is improved but still pragmatic

Status and plan logging are now better, but still primarily diagnostic rather than UX-perfect.

## 4) PATH-based app-server detection is better, not perfect

The app-server availability check now verifies PATH resolution more honestly, but environment-specific oddities are still possible.

## Explicit Next Steps for the Next Developer

## Priority 1 — properly handle multi-question `requestUserInput`

This is the main follow-up we want.

### Desired outcome

Replace the current best-effort combined-question + line-splitting approach with a more structured workflow that preserves:

- one question → one answer mapping
- per-question choices
- per-question freeform allowance
- predictable validation
- a clear user-facing answering flow

### Suggested implementation approach

Start by inspecting:

- `src/providers/codex-adapter.ts`
  - current `item/tool/requestUserInput` handling
  - helper logic for combined question building and answer splitting
- `src/services/question-registry.ts`
  - current registry model supports a single pending question per task
- `src/tools/answer-question.ts`
  - current answer tool assumes one answer string

### Likely design direction

The cleanest long-term answer is probably to represent multi-question prompts explicitly instead of collapsing them into one string.

Possible directions to evaluate:

1. extend `PendingQuestion` to support multiple structured questions
2. add a dedicated multi-answer submission format in `answer-agent`
3. keep backward compatibility for single-question flows
4. continue returning the app-server-required response shape:
   - `{ answers: { [questionId]: { answers: string[] } } }`

### What to test when you implement it

You should verify at least these scenarios:

- single-question freeform answer
- single-question choice answer by number / text
- multi-question freeform answers
- multi-question mixed choice + freeform answers
- timeout behavior
- cancellation behavior
- question cleanup if the Codex session exits mid-question

## Priority 2 — add a more intentional manual question-flow regression script

We currently have:

- `pnpm mcp:smoke`
- the direct interrupt script above

What we **do not** yet have is a repeatable manual flow that deliberately triggers the Codex question path and proves end-to-end answering works under app-server mode.

### Suggested next manual test

Use a Codex prompt that forces a question rather than completing directly. For example, ask Codex to continue only after choosing between several implementation directions, then:

1. launch a Codex-backed task
2. confirm task enters question/input-required state
3. inspect `task:///all` or the task resource
4. answer through `answer-agent`
5. confirm execution resumes and completes

### Files involved in that follow-up

- `src/providers/codex-adapter.ts`
- `src/services/question-registry.ts`
- `src/tools/answer-question.ts`
- `scripts/mcp-stdio-smoke.mjs` or a new targeted script if you add one

## Priority 3 — harden unknown app-server request handling

As upstream evolves, revisit the default fallback path for unknown server requests and add explicit handling where schemas require it.
